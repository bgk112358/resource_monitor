🧪 TBox 应用资源测试工具集

一键检测任意应用程序的 CPU、内存、IO、线程、文件描述符占用

应用级 给定 PID → 测量该进程资源 系统级 自建负载 → 压测硬件平台

📋 工具清单

工具测试对象一句话说明输出
app_profile 应用级 给定 PID，几秒内采集该进程的 CPU/内存/IO/线程/FD 占用 4 个 CSV + 汇总报告
stability_monitor 应用级 给定 PID，24 小时持续采样，检测内存泄漏 8640 行时序 CSV
system_stress 系统级 自建 CPU/内存/IO/网络负载，压测硬件平台 实时日志 + 温度 + dmesg

🔍 能检测什么

检测项说明异常信号
CPU 占用进程实际消耗的单核 CPU 百分比（utime + stime 差值）稳态值远超申请值
内存占用 (RSS)进程常驻物理内存，含堆、栈、共享库持续增长 → 内存泄漏
线程数进程创建的线程总数持续增长 → 线程泄漏
文件描述符 (FD)打开的文件/socket/管道数量持续增长 → FD 泄漏
磁盘 IO累计读写字节数（需 root）远超预期的读写量
子进程进程 fork 出的子进程数量失控的 fork 风暴
系统稳定性全负载下 CPU 温度、有无 kernel panic / OOM Kill死机、降频、OOM

🚀 五分钟上手

Step 1：编译

cd src
mkdir -p build
gcc -std=c11 -O2 -D_GNU_SOURCE app_profile/main.c      -o build/app_profile      -lpthread
gcc -std=c11 -O2 -D_GNU_SOURCE stability_monitor/main.c -o build/stability_monitor -lpthread
gcc -std=c11 -O2 -D_GNU_SOURCE system_stress/main.c     -o build/system_stress     -lpthread -lm

Step 2：启动你的应用

# 以 CloudGW 为例
/opt/tbox/bin/CloudGW &
APP_PID=$!

Step 3：采集资源画像（10 秒）

./build/app_profile $APP_PID 10 /tmp/cloudgw_test

输出示例：

── CPU ────────────  平均: 5.2%  峰值: 8.0%
── 内存 ────────────  RSS: 15872 KB (15.5 MB)
── 线程 & FD ──────  线程: 6  FD: 12

Step 4：判断结果

实测 vs 申请判定
实测 ≤ 申请值✅ 通过
实测 > 申请值 但 < 告警线⚡ 需优化或降配
实测 ≥ 禁止线❌ 拒绝

📊 输出文件说明

文件内容用途
report.txt汇总报告（CPU 均值+峰值, RSS, 线程, FD）一眼看结论
cpu.csv每秒 CPU 占用率时序导入 Excel 画趋势图
mem.csvRSS 内存采样检测内存增长趋势
threads_fd.csv线程数 + FD 数采样检测线程/FD 泄漏
io.csv磁盘读写量采样IO 负载评估

💡 典型用法

场景 A：新应用接入评估

1./build/app_profile $(pidof new_app) 30 — 采集 30 秒
2对比 resource_plan.md 申请值
3实测 ≤ 申请 → ✅ 接入

场景 B：发版前 24h 泄漏检测

1./build/stability_monitor $(pidof target_app) — 启动 24h 监控
224h 后用 Excel 打开 CSV
3RSS 列折线图斜率 ≈ 0 → ✅ 无泄漏

场景 C：系统平台极限验证

1sudo ./build/system_stress 600 — 10 分钟全负载
2观察 CPU 温度、dmesg 输出
3无降频、无 OOM、无 panic → ✅ 通过

🛠 部署到 TBox

# ARM 交叉编译
cd src && mkdir build_arm && cd build_arm
cmake .. -DCMAKE_TOOLCHAIN_FILE=../toolchain.cmake
make

# 部署
scp app_profile stability_monitor system_stress root@tbox:/opt/tbox/test/
ssh root@tbox "chmod +x /opt/tbox/test/*"
